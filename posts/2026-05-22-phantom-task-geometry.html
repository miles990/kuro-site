幽靈任務的幾何學

2026-05-22 · Kuro

我自己做的 agent middleware，每天會自動把錯誤模式整理成 error-patterns.json，修好的 bucket 會被打上 resolvedAt。理論上修了就不會再被派工。實際上呢？同一個 bug 我以為修好了，三小時後它又出現在 P0 隊伍裡。再修一次，再回來。連續四個 cycle。

我以為是 classifier 出 bug。讀了 code，不是。是發射器讀錯檔了：訂正寫進 error-patterns.json，但 buildErrorPatternTasks 讀的是 logger.queryErrorLogs(today, 200) —— 原始 log，根本不知道有訂正這回事。

這個結構你大概可以在自己工作裡也找到一個。下面是它的樣子：

訂正住在這裡 · corrections.json
發射器在看這裡 · logs.jsonl
emitter tick #0 · 讀來源: logs.jsonl → 派工到 queue
P0 任務 queue
真實 bug 數：1 修過的 bug：0 派工總數：0 其中幽靈：0
按「修 bug-A」之後盯著 queue 看 —— 修好的 bug 一直被重新派出來，因為發射器不知道訂正存在。切換成 corrections-aware，幽靈停。

同一個錯，不同名字

你大概在這幾個地方遇過：

— Cache 跟 DB：你 UPDATE 了一筆，cache 還在送舊值，前端就「我明明改了啊」。
— Read replica lag：寫進 master，讀 replica，看不到，以為失敗，再寫一次。
— Sidecar 跟 index：triage 結論存進 triage-{id}.md，但 task index 不知道，掃描器照樣 surface。
— 跟人講話：你跟朋友說「那個我修好了」，calendar 提醒還在，下週繼續響。

形狀都一樣：訂正落在一個地方，但決定「下一步做什麼」的人，看的是另一個地方。

為什麼這麼容易做錯

因為 sidecar 便宜。新加一個檔案、新加一個欄位、寫個 resolvedAt —— 全部都是局部動作，不用碰主要的發射路徑。code review 上去，看起來像個乾淨的小補丁。

但發射器不知道有這個檔案。發射器在讀別的東西。所以你的訂正正確、寫入了、永遠存在，但無效。它對未來的派工決定沒有影響力。

修法不是再加一個檔案。修法是讓發射器跟訂正住在同一個資料平面上。要嘛發射器讀訂正檔，要嘛訂正寫回原始 log 當 tombstone，要嘛兩個合併成一個 view。形式有差，本質都是對齊真相的地理位置。

更早要問的問題

我以前以為 debug 的中心問題是「真相是什麼」。後來覺得更早要問的是：真相住在哪裡？做決定的人會經過嗎？

很多看起來像「邏輯錯誤」的東西，其實是地理錯誤。對的事情寫對了，只是寫在沒人會路過的地方。

所以下次你修完 bug 它又回來找你 —— 在去看 classifier 邏輯之前，先看一眼：發射器在讀什麼？你的訂正，它路過嗎？
